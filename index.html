Nascimento do D — Declaração e Inicialização

Linha 564 — Declaração:

let D, usuAtual = null;

D é declarado como variável global no escopo do módulo principal, sem valor inicial (undefined). Não é um const e não é atribuído ao window, mas o escopo global do script o torna acessível em toda a aplicação.

Linha 576 — Estrutura canônica (ddp()):

function ddp() {
  return {
    alunos:       [],
    presencas:    {},
    pagamentos:   {},
    livros_ent:   {},
    provas:       {},
    notas:        {},
    livros_curso: [{ id: 'l1', nome: 'Livro 1...' }, ...],
    aulas_log:    {}
  };
}

Esta função retorna a estrutura vazia canônica de D. É usada como fallback em carregarDB() e como reset em apagarTudo().

Linha 578 — Primeira população (carregarDB):

⚠ Conflito silencioso confirmado — D.pagamentos D.pagamentos é populado do localStorage (com dados históricos hardcoded), mas nunca sincronizado com o Supabase. A tela de perfil de aluno (abrirAluno) ainda lê D.pagamentos para calcular histórico de pagamentos. O dashboard (renderDashAdmin) ignora D.pagamentos e vai direto ao Supabase — isso cria divergência entre a visão do dashboard e a visão do perfil individual.
⚠ D sobrescreve Supabase ou Supabase sobrescreve D? Para presencas e alunos: Supabase sobrescreve D completamente (sem merge). Para notas/livros/provas: Supabase sobrescreve D bloco a bloco — o que significa que dados locais de blocos não presentes no Supabase sobrevivem silenciosamente. Para pagamentos: D nunca é sobrescrito pelo Supabase (ele nunca é carregado de lá).

Quando salvar() é chamado:

  • Após salvarAluno() e delAluno()
  • Após salvarPres() (presença manual)
  • Após _setLivroEnt() e _setProvaEnt()
  • Após _salvarNotaAluno() e salvarNotas()
  • Após salvarEdicaoAluno()
  • Após carregarPresencaSupabase() no fluxo de aluno
  • Em apagarTudo()

Não há debounce em salvar(). Cada operação grava o JSON completo de D no localStorage imediatamente. Em sessões longas ou com muitos alunos, isso pode gerar escritas grandes frequentes.

⚠ Risco de stale cache confirmado Quando o sistema é carregado e inicializarApp() termina de carregar o Supabase, as funções que sobrescrevem D (alunos, notas, presencas) não chamam salvar() na rota admin. Isso significa que o localStorage permanece com dados antigos até a próxima operação que chame salvar(). Se a página for recarregada antes de qualquer operação, D será populado com dados do localStorage (não do Supabase) — potencialmente stale.
ℹ Exceção: fluxo de aluno chama salvar() Na rota de aluno (inicializarApp aluno), após carregarPresencaSupabase(), há um salvar() explícito (linha 7178). Isso significa que para alunos o localStorage é atualizado pós-sync. Para admins, não.

07

Race Conditions

Race 1 — renderDashAdmin vs renderMetricasMaterial em irAdmin('inicio'):

// irAdmin('inicio') — linhas 2816-2817
if(t==='inicio') renderDashAdmin().catch(...); // async, sem await
if(t==='inicio') renderMetricasMaterial();    // síncrono, executa imediatamente

As duas funções disparam quase simultaneamente. renderMetricasMaterial usa D (que pode estar desatualizado). renderDashAdmin busca dados frescos no Supabase. Não há coordenação entre elas.

Race 2 — Loaders sequenciais vs Render que usa D:

// inicializarApp — rota admin
await carregarAlunosSupabase();   // D.alunos = [...Supabase]
await carregarLivrosEntSupabase(); // D.livros_ent atualizado
await carregarProvasEntSupabase(); // D.provas atualizado
await carregarNotasAdminSupabase(); // D.notas atualizado
await renderDashAdmin();           // Consulta Supabase diretamente
irAdmin('inicio');               // Chama renderMetricasMaterial (usa D)

Neste fluxo, os loaders são sequenciais (await), então renderMetricasMaterial recebe D atualizado. O risco aqui é mínimo para o boot via restauração de sessão.

Race 3 — Login manual (loginAdmin) vs D stale:

// loginAdmin — linha 2723
await renderDashAdmin(); // Supabase direto — OK
irAdmin('inicio');      // renderMetricasMaterial usa D do localStorage — STALE
// Os loaders (carregarAlunosSupabase etc.) NÃO são chamados aqui

Quando o admin faz login manualmente (não por restauração de sessão), os loaders de D não são acionados. renderMetricasMaterial exibirá dados do localStorage potencialmente antigos.

Race 4 — salvarNotas com Promise.all:

const ops = [];
ativos().forEach(a => {
    D.notas[pid][a.id] = nota;
    ops.push(sincronizarNotaSupabase(a.id, pid, nota)); // async
});
await Promise.all(ops); // paralelo — sem problema estrutural
salvar();               // localStorage após Promise.all — correto

Esta função não tem race condition — D é mutado de forma síncrona antes do Promise.all, e salvar() é chamado após todas as operações resolverem.

Race 5 — Timer de auto-refresh (60s):

window._renderDashAdminTimer = setTimeout(renderDashAdmin, 60000);

O timer re-executa renderDashAdmin a cada 60 segundos com dados frescos do Supabase. Se o usuário estiver no meio de uma edição (ex: digitando uma nota), o render do dashboard pode conflitar visualmente com a edição em andamento, mas não conflita em D porque renderDashAdmin não usa D.

⚠ Race mais crítico: loginAdmin + renderMetricasMaterial O Race 3 é o mais perigoso em uso real, porque o fluxo de login manual é o mais comum para o admin ao longo do dia. Neste caminho, renderMetricasMaterial sempre executará com D do localStorage, potencialmente stale desde a última sessão.

Impacto nos KPIs Material

renderMetricasMaterial() (linha 3273) calcula e exibe 4 KPIs: Provas Realizadas, Provas Pendentes, Média Geral e Progresso do Curso. Cada um tem uma fonte de dado diferente:

KPI Fonte em D Quando pode ser stale Severidade
Provas Realizadas D.provas Após login manual; blocos locais não sobrescritos pelo Supabase se ausentes Médio
Provas Pendentes D.provas (derivado) Mesma condição de Provas Realizadas Médio
Média Geral D.notas Após login manual; mais exposta pois notas mudam frequentemente Alto
Progresso do Curso D.livros_ent Blocos locais com chaves numéricas sobrevivem se não existirem no Supabase Alto

renderMetricasMaterial possui guards, mas não retry nem invalidação:

// Guard 1: D não inicializado ou sem alunos
if (!D || !Array.isArray(D.alunos)) return;
// Guard 2: container ausente no DOM
var el = document.getElementById('a-metricas-material');
if (!el) return;

Os guards protegem contra crash, mas não garantem que os dados em D sejam frescos. Se D existir mas estiver stale, a função executará normalmente e exibirá os dados antigos sem qualquer aviso.

⚠ Ausência de mecanismo de invalidação renderMetricasMaterial não possui: retry, invalidação de cache, timestamp de última atualização, indicador visual de dados potencialmente desatualizados, ou dependência de sync concluído. Ela simplesmente lê D no momento em que é chamada e renderiza.
ℹ Auto-atualização: não existe Diferentemente de renderDashAdmin (que tem timer de 60s), renderMetricasMaterial não é chamada automaticamente após a carga inicial. Ela só é re-executada quando o admin navega manualmente para a aba de início (irAdmin('inicio')) ou de material.

10

Conclusão Arquitetural Final

Diagnóstico da Arquitetura Real

O sistema opera com dois mundos paralelos de dados que raramente se sincronizam de forma completa e bidirecional.

  • Mundo 1 — Supabase Live: renderDashAdmin, tela de pagamentos e relatórios. Sempre frescos. Não dependem de D.
  • Mundo 2 — D (estado em memória): renderMetricasMaterial, telas de presença e de alunos. Frescos apenas quando os loaders foram executados (boot via sessão). Stale em login manual ou após operações locais.
  • Mundo 0 — localStorage: fonte de nascimento de D no boot. Potencialmente stale do ponto de vista do Supabase. Atualizado assincronamente por operações locais.

Os Três Riscos Estruturais

  • Risco 1 — D.pagamentos nunca sincronizado: a tela de perfil do aluno exibe dados históricos hardcoded ou do localStorage como histórico de pagamentos. O Supabase tem os dados reais em aluno_livros, mas D.pagamentos nunca os recebe.
  • Risco 2 — Login manual não dispara loaders: qualquer sessão iniciada via loginAdmin() (não restauração automática) opera com D do localStorage. renderMetricasMaterial exibirá dados potencialmente da sessão anterior.
  • Risco 3 — Operações locais sem sync reverso: salvarAluno, delAluno, _setLivroEnt e _setProvaEnt escrevem em D e localStorage, mas nunca notificam o Supabase. Na próxima carga dos loaders, o Supabase sobrescreve esses dados locais silenciosamente.

Resumo Executivo

A divergência arquitetural central identificada na auditoria funcional anterior (renderDashAdmin = Supabase live vs renderMetricasMaterial = D em memória) é real e confirmada. A causa raiz é a ausência de um único canal de verdade: parte do sistema foi migrada para queries diretas ao Supabase, enquanto outra parte permanece dependente de D — que é alimentado por loaders que não são chamados em todos os fluxos de entrada.

O estado global D funciona hoje como um cache semi-manual: é populado no boot por restauração de sessão, mas não é invalidado, não tem TTL, não tem indicador de staleness, e não é atualizado quando o Supabase muda por ação de outro componente.

Qualquer evolução futura dos KPIs acadêmicos que dependa de renderMetricasMaterial herdará todos esses riscos de stale state enquanto a arquitetura de D permanecer inalterada.
